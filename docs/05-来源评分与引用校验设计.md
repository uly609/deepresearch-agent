# 来源评分与引用校验设计

这篇讲 `deepresearch/agents.py` 里的两个 Agent：

- `SourceEvaluatorAgent`
- `CitationVerifierAgent`

## 为什么需要来源评分

DeepResearch 不是搜到什么都信。

不同来源可信度不一样：

- 官方文档通常更权威
- 论文适合支持架构和趋势分析
- GitHub 适合观察生态和实现
- 博客和新闻只能作为旁证

所以要评分。

## 当前评分维度

`SourceEvaluatorAgent` 当前按四个维度评分：

```text
authority   权威性
freshness   新鲜度
relevance   相关性
risk        风险
```

最后得到：

```text
final
```

这个分数会进入报告。

相关性不是只看关键词。当前流程是在 `dedupe` 后先由 `EvidenceRetriever.index_sources()` 建 EvidenceChunk 和向量索引，然后 `score` 节点调用：

```python
runtime.evaluator.score(research_state.sources, question, runtime.rag)
```

Evaluator 会从 RAG 里取 `source_similarity()`，把向量相似度和关键词/LLM 语义判断融合成 relevance。没有 LLM 时用规则兜底；开启 LLM 时，模型会一起判断 authority、freshness、relevance、risk。

## 为什么需要引用校验

Agent 最容易出的问题是：

> 结论写得很像真的，但来源并不支持。

所以需要 `CitationVerifierAgent`。

当前版本会把每个来源绑定到最相关的 EvidenceChunk，再生成 claim，并标记：

```text
supported
weak
unsupported
```

如果开启 LLM，`CitationVerifierAgent` 会让模型直接判断 claim 是否被 evidence 支撑；没有 LLM 时用来源分数和证据关键词重合兜底。报告生成后，`ReportAuditorAgent` 还会做逐句审计：

```text
报告中的关键句
-> 找 supported claim
-> 判断是否 grounded / weak / uncited
-> 把审计结果写进报告
```

## 面试话术

可以这样讲：

> 我把来源评分和引用校验拆成两个 Agent。Evaluator 负责从 authority、freshness、relevance、risk 四个维度评估来源质量，其中 relevance 会融合 RAG 向量相似度；Verifier 负责把 claim 绑定到 EvidenceChunk，判断结论是否有证据支持。开启 LLM 后，来源评分、引用验证和报告审计都会由模型辅助判断；没有 LLM 时用规则和向量检索兜底。这样报告不是单纯由模型生成，而是尽量让每个关键结论都能追溯到来源。

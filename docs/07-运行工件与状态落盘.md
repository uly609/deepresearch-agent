# 运行工件与状态落盘

这篇讲 `deepresearch/run_store.py` 和 `deepresearch/task_state.py`。

## 为什么要保存运行工件

Agent 项目不能只留下最终答案。

你还需要知道：

- 任务 ID 是什么
- 原始问题是什么
- 拆了哪些子问题
- 搜到了哪些来源
- 每个来源分数是多少
- 生成了哪些 claim
- 过程里发生了哪些事件
- 最终报告是什么

这些就是运行工件。

## RunStore 保存什么

当前保存到：

```text
.deepresearch/runs/{run_id}/task_state.json
.deepresearch/runs/{run_id}/trace.jsonl
.deepresearch/runs/{run_id}/research_state.json
.deepresearch/runs/{run_id}/report.md
.deepresearch/runs/{run_id}/report.json
.deepresearch/runs/{run_id}/checkpoint.json
```

`task_state.json` 保存这次 run 进行到哪一步。

`trace.jsonl` 保存过程事件。

`research_state.json` 保存计划、来源、评分、claim 和事件。

`report.md` 保存最终报告。

`checkpoint.json` 保存后续恢复需要的轻量锚点。

## 为什么 state 和 report 分开

因为它们用途不同。

`task_state.json` 和 `trace.jsonl` 用来复盘和调试：

```text
Agent 每一步做了什么
中间状态是什么
来源和分数是什么
```

`report.md` 用来给用户看：

```text
最终研究报告
```

## 面试话术

可以这样讲：

> 我没有只保存最终报告，而是参考 Pico 的运行工件设计，把单次 run 拆成 `task_state.json`、`trace.jsonl`、`research_state.json`、`report.md` 和 `checkpoint.json`。`task_state` 表示当前运行状态，`trace` 用于复盘过程，`research_state` 保存结构化研究证据，`report` 给用户看，`checkpoint` 为后续恢复做准备。

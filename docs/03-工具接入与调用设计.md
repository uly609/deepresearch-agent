# 工具接入与调用设计

这篇讲 `deepresearch/tools.py`。

## 这篇解决什么问题

Agent 不能只靠模型自己回答，它需要工具。

DeepResearch 方向常见工具包括：

- GitHub Search
- arXiv Search
- Web Search
- RAG Retriever
- MCP Tools
- 文件解析
- 浏览器工具

但 workflow 不应该直接依赖某一个具体工具。

所以项目里先抽象了：

```python
class SourceConnector
```

## SourceConnector 是什么

它是所有搜索来源的统一接口。

现在要求所有 Connector 都实现：

```python
def search(self, query: str) -> List[Source]
```

这样以后不管你接什么工具，都统一返回 `Source`。

比如未来可以有：

```text
GitHubSearchConnector
ArxivConnector
WebSearchConnector
McpSearchConnector
RagConnector
```

workflow 不需要知道底层细节。

## 当前 Connector 有哪些

离线模式默认用这些：

- `FrameworkConnector`
- `OfficialDocsConnector`
- `DeepResearchExampleConnector`
- `PaperConnector`
- `SafetyConnector`

它们不是实时联网搜索，而是内置固定来源，先保证 Agent 工作流可复现。

开启 live tools 后会增加：

- `GitHubSearchConnector`
- `ArxivSearchConnector`
- `WebSearchConnector`
- `MCPSearchConnector`
- `BraveSearchConnector`
- `TavilySearchConnector`

其中 `MCPSearchConnector` 读取环境变量 `MCP_SEARCH_ENDPOINT`，向 MCP 搜索网关发请求：

```json
{"tool": "search", "query": "...", "limit": 5}
```

如果 MCP 服务没配置或调用失败，它会返回 diagnostic 结果，`FallbackConnector` 会自动切回离线来源。

为什么先这么做？

因为第一阶段要先验证 Agent 工作流，而不是一开始就被外部 API、网络、鉴权卡住。

面试时要诚实：

> 当前版本离线模式用内置 Connector 保证链路稳定；live 模式已经预留 GitHub、arXiv、Web Search 和 MCP Search Connector。工具层统一返回 Source，所以主 workflow 不需要关心底层是本地索引、HTTP API 还是 MCP Gateway。

## ToolRegistry 是什么

`ToolRegistry` 管理多个 Connector。

核心方法是：

```python
def search_all(self, query: str) -> List[Source]:
```

它会调用所有 connector，然后按规范化 URL 去重。规范化会去掉 fragment 和 `utm_*` 等跟踪参数，避免同一个来源因为链接形态不同被重复保存。

同时它会给每个 `Source.metadata` 写入检索 provenance：

```text
retrieved_query  这条来源是由哪个 query 搜出来的
retrieved_by     哪个 connector 返回的
connectors       多个 connector 命中同一 URL 时的来源列表
canonical_url    规范化后的 URL
```

这些字段用于追踪和复盘，但不会进入 RAG 的 evidence text，避免把“搜索 query”误当成来源证据。

这相当于 Agent 的工具注册中心。

## 面试话术

可以这样讲：

> 我没有让 workflow 直接调用具体 API，而是抽象了 `SourceConnector`。所有外部检索工具都实现统一的 `search(query)` 接口，并返回统一的 `Source` 数据结构。搜索前会先由 Query Planner 把子问题改写成多条 query，`ToolRegistry` 再统一调度 GitHub、arXiv、Web、MCP、Brave、Tavily 等 Connector，并记录每条来源的检索 provenance。这样后续换搜索服务或 MCP Gateway，不需要改主 workflow。

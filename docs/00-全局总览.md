# DeepResearch Agent 全局架构导览

这篇先帮你把整个项目看成一条能跑通的 Agent 执行链。

如果你是第一次看这个项目，读完以后至少要能回答这些问题：

- 这个项目到底是什么，不是什么
- 一条研究问题进来以后，系统怎么一步步往前推
- 为什么它不是普通问答，也不是普通后端
- `langgraph_loop`、`tools`、`context_manager`、`eval_harness` 分别解决什么问题
- 现在已经做到哪里，后面还差什么

## 一句话定义

`DeepResearch Agent` 是一个 Python-first 的研究型 Agent 原型。

它要解决的问题不是“用户问一句，模型答一句”，而是：

> 把一个开放研究问题，推进成一条有计划、有工具、有上下文、有引用、有评估的执行链。

当前版本已经有 LangGraph 主流程、真实/离线工具、可选 LLM、轻量 RAG 和本地向量库。它现在的定位是：

> Agent 核心工作流已经跑通，真实工具和模型能力可选启用，生产级检索和逐句引用校验仍可继续增强。

## 先看三条主线

这个项目可以先按三条线理解。

| 主线 | 解决的问题 | 对应文件 |
| --- | --- | --- |
| 控制流 | 一条请求怎么被一步步推进 | `run_agent.py`、`deepresearch/cli.py`、`deepresearch/runtime.py`、`deepresearch/langgraph_loop.py` |
| 能力层 | 每个 Agent / 工具具体干什么 | `deepresearch/agents.py`、`deepresearch/tools.py`、`deepresearch/security.py`、`deepresearch/llm_provider.py` |
| 状态和证据 | 过程怎么保存，结果怎么评估 | `deepresearch/rag.py`、`deepresearch/vector_store.py`、`deepresearch/task_state.py`、`deepresearch/run_store.py`、`deepresearch/checkpoint.py`、`deepresearch/eval_harness.py` |

你先不要急着背所有文件。先把这三条线记住。

## 一条请求怎么跑完整个系统

假设你运行：

```bash
make run
```

真实执行的是：

```bash
python3 -m deepresearch
```

系统大概会走下面这条线：

1. `python3 -m deepresearch` 进入 `deepresearch/__main__.py`
2. `cli.py` 解析命令行参数，判断是普通运行还是 eval
3. `cli.py` 创建 `DeepResearchRuntime`
4. `runtime.py` 装配 Planner、Tools、Guard、Evaluator、Verifier、Reporter 和 RunStore
5. `langgraph_loop.py` 创建 `TaskState`、`ResearchState` 和 `ContextManager`
6. `PlannerAgent` 把大问题拆成多个子问题
7. `ToolRegistry` 调用 GitHub、arXiv、Web、官方文档或离线 Connector 搜索候选来源
8. `PromptInjectionGuard` 过滤可疑来源内容
9. `langgraph_loop` 对来源按 URL 去重，并用 RAG/向量库建立 EvidenceChunk
10. `SourceEvaluatorAgent` 给来源打分
11. `CitationVerifierAgent` 把来源转成可验证 claim，并绑定 evidence excerpt
12. `ConflictDetectorAgent` 检查来源之间的支持强弱冲突
13. `ReflectionAgent` 检查是否缺少 RAG、MCP、context、safety、evaluation 等证据
14. `ReportAgent` 生成 Markdown 报告
15. `RunStore` 保存 `task_state.json`、`trace.jsonl`、`research_state.json`、`report.md`、`checkpoint.json` 和 `vector_store.json`

压成一句话：

> CLI 接收问题，Runtime 装配运行现场，LangGraph 推进流程，Agents 完成各阶段能力，RunStore 留下可复盘的运行工件。

## 当前项目做到什么地步

已经做到：

- Python 项目主结构已经建立
- 普通运行可以生成报告
- Eval 模式可以跑固定题集
- Agent 流程有事件轨迹
- 来源有去重、评分、引用校验
- 有上下文管理雏形
- 有 Prompt Injection Guard 雏形
- 有运行工件：`task_state.json`、`trace.jsonl`、`research_state.json`、`report.md`、`checkpoint.json`、`vector_store.json`

后续还可以增强：

- 更稳定的商业 Web Search Provider
- 把本地哈希向量库替换为 FAISS、Chroma、Milvus 或 pgvector
- 报告逐句引用校验
- MCP Client / MCP Gateway
- 完整 GraphState checkpoint resume

## 面试时怎么总览介绍

可以这样讲：

> 我做的是一个 Python-first 的 DeepResearch Agent 原型。结构上把入口、runtime、LangGraph 工作流、task state、run store、checkpoint、RAG 和 eval 分开。功能上不是本地代码问答，而是 DeepResearch：计划、检索、安全过滤、来源评分、证据检索、引用校验、冲突检测、反思和报告生成。当前版本支持离线演示，也可以启用 GitHub、arXiv、Web Search 和 DeepSeek/OpenAI-compatible LLM。

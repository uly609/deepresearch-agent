# 上下文管理设计

这篇讲 `deepresearch/context_manager.py`。

## 为什么 Agent 需要上下文管理

很多新手会以为：

> 把所有历史都塞进 prompt 就行。

但 Agent 跑长任务时，这样一定会乱。

因为上下文里有不同类型的信息：

- 当前任务目标
- 最近步骤
- 搜到的来源
- 已经验证过的 claim
- 可以丢弃的旧日志
- 不能丢的关键证据

这些不能混在一起。

## ContextManager 管什么

当前 `ContextManager` 管这些字段：

```text
task_goal          当前任务目标
working_notes      工作记忆
evidence_pool      证据池
verified_claims    已验证结论
discarded_notes    被丢弃的旧笔记
max_context_chars  单次上下文预算，默认 6000 字符
max_evidence_items 最多保留的来源条数，默认 8 条
```

你可以把它理解成 Agent 的资料夹。

## evidence_pool 为什么重要

DeepResearch 最重要的是证据。

如果模型生成了一个结论，但找不到来源支持，就容易幻觉。

所以项目里把来源集中放进：

```python
evidence_pool
```

后面引用校验、报告生成、上下文压缩都可以围绕证据池做。

## compact() 是什么

`compact()` 会把上下文压成一段短文本。

它会先按 `SourceScore.final` 选择高分来源，再按字符预算截断；来源超过上限不会删除，而是留在 evidence pool，记录为 deferred evidence。

当前已经把压缩后的上下文输入给 Reflect 和 Report 的 LLM prompt：

```text
目标
最近笔记
关键证据
已验证 claim
```

这就是上下文压缩的雏形。

## 面试话术

可以这样讲：

> 我没有把所有历史直接拼进 prompt，而是设计了 `ContextManager`，把任务目标、工作记忆、证据池、已验证 claim 分开管理。评分后只把高分证据压缩进固定字符预算，并提供给 Reflect 和 Report 节点；其余证据仍保留在证据池中，需要时可以被 RAG 检索回来。这样既控制上下文窗口，也不会因为截断直接丢掉原始证据。

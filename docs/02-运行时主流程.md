# 运行时主流程设计

这篇讲 `deepresearch/runtime.py` 和 `deepresearch/langgraph_loop.py`。

## 这篇解决什么问题

一条用户研究问题，怎么被推进成一次完整的 Agent 执行？

核心分两层：

- `DeepResearchRuntime`：装配运行现场
- `LangGraphAgentLoop`：用 LangGraph 节点和边推进流程

## 主流程

入口代码在：

```python
class DeepResearchRuntime:
    def ask(self, question: str, emit=...):
        return self.loop.run(question, emit=emit)
```

当前主工作流在：

```python
class LangGraphAgentLoop:
```

它把流程注册成节点：

```text
start
-> plan
-> search
-> dedupe
-> score
-> verify
-> reflect
-> report
```

其中 `reflect` 后面是条件边：

```text
有 gaps 且未超过搜索轮数 -> search
没有 gaps 或到达上限 -> report
```

## 每个节点调用谁

```text
plan
  -> PlannerAgent.plan()

search
  -> SearchQueryPlannerAgent.expand()
  -> ToolRegistry.search_all()
  -> PromptInjectionGuard.filter_sources()

dedupe
  -> runtime.dedupe_sources()
  -> EvidenceRetriever.index_sources()
  -> LocalVectorStore.upsert()

score
  -> SourceEvaluatorAgent.score()
  -> EvidenceRetriever.source_similarity()

verify
  -> CitationVerifierAgent.verify()
  -> EvidenceRetriever.best_for_source()
  -> ConflictDetectorAgent.detect()

reflect
  -> ReflectionAgent.find_gaps()

report
  -> ReportAgent.write()
  -> RunStore.write_report()
  -> RunStore.write_checkpoint()
```

## 为什么用 LangGraph

手写流程适合线性 demo，但 DeepResearch 更需要状态图：

- 任务有多个阶段
- 每个阶段都要读写 `GraphState`
- 反思后可能回到搜索
- 长任务需要 trace、checkpoint 和 resume

所以现在只保留 LangGraph 主线，避免两套流程同时存在。

## 面试话术

> `runtime.py` 负责装配 Planner、工具、RAG、Evidence Graph、安全层、评分器、引用校验器、冲突检测器、报告器、RunStore 和 AgentHarness；`langgraph_loop.py` 负责把一次研究任务映射成 LangGraph 节点。流程为 plan、query_rewrite、search、dedupe、rag、graph_rag、score、verify、reflect 到 report。`AgentHarness` 会包装每个节点，记录节点耗时、状态、phase 和失败原因；`RunPolicy` 控制每轮 query 数、工具调用步数、来源数和补搜轮次。`query_rewrite` 负责把子问题或信息缺口展开为多个检索表达；`rag` 专门把去重来源建立为 EvidenceChunk，并用 BM25 + Embedding + RRF 做融合检索；`graph_rag` 从证据片段里抽取实体和共现关系。每一步都会更新 `ResearchState` 和 `TaskState`，并通过 `RunStore` 落盘 trace、report、checkpoint、vector store、evidence graph 和 harness steps。`reflect` 节点后使用条件边，如果发现信息缺口会回到 `query_rewrite` 再补搜，否则进入 report。

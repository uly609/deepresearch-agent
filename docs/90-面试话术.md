# DeepResearch Agent 面试话术

这份文档帮你把项目压成能讲出口的话。

## 30 秒介绍

> 我做的是一个 Python-first 的 DeepResearch Agent 原型。结构上参考 Pico 的 harness，把 runtime、agent loop、task state、run store、checkpoint、context manager 和 eval 分开；功能上换成 DeepResearch，用户输入研究问题后，系统会拆解子问题、调用检索工具、过滤风险来源、去重、评分、引用校验、反思并生成 Markdown 报告。当前版本先用内置 Connector 跑通 Agent 工作流，后续会接真实 GitHub、arXiv、Web Search、RAG 和 MCP。

## 1 分钟介绍

> 这个项目的重点不是普通后端，而是 Agent 执行链路。我参考 Pico 的实现方式，把入口、运行时装配、主循环、任务状态、运行工件和 checkpoint 拆开。DeepResearch 的业务阶段包括 Planner 生成研究计划，ToolRegistry 调用检索工具，ContextManager 管理工作记忆和证据池，SourceEvaluator 对来源做权威性、新鲜度、相关性和风险评分，CitationVerifier 把结论和来源绑定，ReflectionAgent 检查信息缺口，ReportAgent 生成最终报告。整个过程会写入 task_state、trace、research_state 和 report，方便复盘和评估。

## 如果问：现在有没有真实搜索？

可以答：

> 当前版本还没有接真实搜索 API。现在的 `BuiltInResearchConnector` 是模拟外部检索，用于先验证 Agent 工作流、数据结构、事件轨迹、来源评分和引用校验。因为工具层已经抽象成 `SourceConnector`，后续替换成 GitHub Search、arXiv、Web Search 或 MCP Connector 时，不需要改主 workflow。

## 如果问：为什么用 Python？

可以答：

> 我这个项目核心是 Agent，不是普通 Java 后端。Agent 方向里，LangGraph、CrewAI、AutoGen、RAG、浏览器工具、Eval Harness 和 MCP 生态主要集中在 Python，所以核心执行链路用 Python 更合理。Java 版本我已经归档，只保留为早期原型参考。

## 如果问：Agent 能力体现在哪里？

可以答：

> 它不是一次问答，而是一条多阶段执行链。系统会先拆解任务，再调用工具获取来源，然后对来源做安全过滤、去重、评分和引用校验，还会用 ReflectionAgent 检查信息缺口，最后生成可追溯报告。这个过程有状态、有事件、有运行工件，也有 Eval Harness 评估，不只是一个 prompt 模板。

## 如果问：下一步怎么优化？

可以答：

> 我会按四步优化。第一步接真实工具，包括 GitHub、arXiv 和 Web Search。第二步接 LLM，让 Planner 和 Reporter 从规则版升级成模型版，并要求结构化 JSON 输出。第三步加 RAG 和上下文压缩，把论文、官方文档和历史报告作为可检索证据池。第四步加强 Eval Harness，评估引用准确率、来源多样性、工具成功率和失败原因。

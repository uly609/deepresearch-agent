# DeepResearch Agent 面试话术

这份文档帮你把项目压成能讲出口的话。

## 30 秒介绍

> 我做的是一个 Python-first 的 DeepResearch Agent 原型。结构上参考 Pico 的 harness，把 runtime、LangGraph workflow、task state、run store、checkpoint、context manager 和 eval 分开；功能上换成 DeepResearch，用户输入研究问题后，系统会拆解子问题、调用 GitHub/arXiv/Web/MCP 等检索工具，过滤风险来源，去重后建立 EvidenceChunk 和向量索引，再做来源评分、引用校验、反思补搜并生成 Markdown 报告。

## 1 分钟介绍

> 这个项目的重点不是普通后端，而是 Agent 执行链路。我参考 Pico 的实现方式，把入口、运行时装配、工作流、任务状态、运行工件和 checkpoint 拆开。DeepResearch 的业务阶段包括 Planner 生成研究计划，ToolRegistry 调用检索工具，RAG 把来源切成 EvidenceChunk 并做混合检索，SourceEvaluator 对来源做权威性、新鲜度、相关性和风险评分，CitationVerifier 把结论和证据绑定，ReflectionAgent 检查信息缺口，ReportAgent 生成最终报告。整个过程会写入 task_state、trace、research_state、vector_store 和 report，方便复盘和评估。

## 如果问：现在有没有真实搜索？

可以答：

> 当前版本已经把工具层抽象成 `SourceConnector`，默认会尝试 GitHub、arXiv、Web 和 MCP 搜索，失败时自动回退到内置资料，保证演示不断掉。这样主 workflow 只调用 `ToolRegistry.search_all()`，后面替换成 Tavily、Brave、Serper 或真实 MCP Gateway 时，不需要改 LangGraph 主流程。

## 如果问：为什么用 Python？

可以答：

> 我这个项目核心是 Agent，不是普通 Java 后端。Agent 方向里，LangGraph、CrewAI、AutoGen、RAG、浏览器工具、Eval Harness 和 MCP 生态主要集中在 Python，所以核心执行链路用 Python 更合理。Java 版本我已经归档，只保留为早期原型参考。

## 如果问：Agent 能力体现在哪里？

可以答：

> 它不是一次问答，而是一条多阶段执行链。系统会先拆解任务，再调用工具获取来源，然后对来源做安全过滤、去重、评分和引用校验，还会用 ReflectionAgent 检查信息缺口，最后生成可追溯报告。这个过程有状态、有事件、有运行工件，也有 Eval Harness 评估，不只是一个 prompt 模板。

## 如果问：下一步怎么优化？

可以答：

> 我会按四步优化。第一步接更稳定的商业搜索，比如 Tavily、Brave 或 Serper。第二步把本地轻量向量库替换成 FAISS、Chroma、Milvus 或 pgvector，并接更强的 embedding/reranker。第三步加强 LLM-as-judge 的批量调用、重试和结构化输出监控。第四步加强 Eval Harness，评估引用准确率、来源多样性、工具成功率和失败原因。

# DeepResearch Agent 面试话术

这份文档帮你把项目压成能讲出口的话。

## 30 秒介绍

> 我做的是一个 Python-first 的 DeepResearch Agent Harness。它不是简单问答，而是把技术调研任务放进一个受控 Agent Runtime 里执行。用户输入问题后，系统会拆解子问题、query rewrite、多源检索、构建 Hybrid RAG 和 Evidence Graph，再做来源评分、引用校验、反思补搜和报告生成。Harness 层负责限制工具步数、搜索轮次和 query fan-out，并记录每个节点的耗时、状态和失败原因，方便复盘和评估。

## 1 分钟介绍

> 这个项目的重点不是普通后端，而是 Agent Harness 的工程化执行链路。我把入口、运行时装配、LangGraph workflow、任务状态、运行工件、checkpoint、context manager 和 eval 分开。业务上，Planner 生成研究计划，Query Rewrite 扩展检索表达，ToolRegistry 调用多源工具，RAG 用 BM25 + Embedding + RRF 做融合排序，GraphRAG 抽取 Evidence Graph，后面再做评分、验证、反思和报告。工程上，AgentHarness 会包装每个节点，记录节点耗时和执行状态；RunPolicy 会限制 query 数、工具调用步数、来源数量和补搜轮次，避免 Agent 长任务失控。最后 task_state、trace、research_state、tool audit、harness steps、vector_store、evidence_graph、结构化 report 和 checkpoint 都会落盘。

## 如果问：现在有没有真实搜索？

可以答：

> 当前版本已经把工具层抽象成 `SourceConnector`，默认会尝试 GitHub、arXiv、Web 和 MCP 搜索，失败时自动回退到内置资料，保证演示不断掉。这样主 workflow 只调用 `ToolRegistry.search_all()`，后面替换成 Tavily、Brave、Serper 或真实 MCP Gateway 时，不需要改 LangGraph 主流程。

## 如果问：为什么用 Python？

可以答：

> 我这个项目核心是 Agent，不是普通 Java 后端。Agent 方向里，LangGraph、CrewAI、AutoGen、RAG、浏览器工具、Eval Harness 和 MCP 生态主要集中在 Python，所以核心执行链路用 Python 更合理。Java 版本我已经归档，只保留为早期原型参考。

## 如果问：Agent 能力体现在哪里？

可以答：

> 它不是一次问答，而是一条多阶段执行链。系统会先拆解任务，再调用工具获取来源，然后对来源做安全过滤、去重、评分和引用校验，还会用 ReflectionAgent 检查信息缺口，最后生成可追溯报告。这个过程有状态、有事件、有运行工件，也有 Eval Harness 评估，不只是一个 prompt 模板。

## 如果问：你说的 Harness 是什么？

> Harness 可以理解成包在 Agent 工作流外面的工程运行层，不负责写具体研究结论，而是负责让 Agent 可控、可观测、可复盘。我的项目里 `AgentHarness` 会包装每个 LangGraph 节点，记录 node、phase、duration、status 和失败原因；`RunPolicy` 会限制每轮 query 数、最大工具调用步数、最大来源数和最大补搜轮次。这样面试官问“Agent 会不会无限循环、工具越调越多、失败怎么排查”时，我不是只说靠 prompt，而是有运行时治理和落盘数据。

## 如果问：上下文窗口如何控制？

> 我没有把所有搜索结果和历史日志都塞进模型。ContextManager 把工作记忆、证据池和已验证 claim 分层保存；评分后优先选择高分来源，控制最多 8 条和 6000 字符上下文，传给 Reflect 和 Report。没有选进 prompt 的来源并不丢弃，仍保留在 evidence pool 和向量库里，后续可以按需检索。

## 如果问：工具调用如何控制和排查？

> 搜索工具通过 ToolRegistry 统一注册，不让工作流任意调用外部函数。Registry 支持 Connector 允许名单、query 长度和单 Connector 返回量限制，并记录每次调用是否成功、耗时、结果数和失败原因。运行结束后这些审计数据会和报告一起落盘，方便排查哪个来源或哪个工具导致结果不稳定。

## 如果问：checkpoint 能否恢复？

> 当前每个关键节点都会刷新 checkpoint，保存 phase、下一步建议和研究进度。当前 CLI 的 resume 采用原问题 replay 方式，目的是任务失败后能重新运行并对比旧工件；完整 GraphState 原地续跑是下一阶段工作，我不会把 replay 说成生产级恢复。

## 如果问：数据怎么提取、如何分片？

> 检索阶段先由 GitHub、arXiv、Web、MCP Connector 把不同返回格式统一为 Source，里面包含标题、URL、摘要、发布时间和来源元信息。开启网页正文抓取后，我会对 Web 和官方文档下载 HTML，去掉 script、style、导航等噪声，把清洗后的正文限制在固定长度后写入 Source。RAG 再把标题、摘要、正文和有效元信息切成带 source_url 的 EvidenceChunk；当前按词窗口 120、重叠 24 切分，既避免单块太长稀释语义，也减少段落边界被切断导致的信息丢失。

## 如果问：它怎么给用户使用？

> 除了 CLI，我提供了 FastAPI 接口。前端通过 POST 创建研究任务，再用 SSE 订阅 plan、search、verify、report 等执行事件，最终通过任务查询接口获取报告。这样长任务不会让一个 HTTP 请求一直阻塞，用户能看到执行过程和失败位置。

## 如果问：下一步怎么优化？

可以答：

> 我会按四步优化。第一步接更稳定的商业搜索，比如 Tavily、Brave 或 Serper。第二步把本地轻量向量库替换成 Milvus/pgvector，把 Evidence Graph 替换成 Neo4j/NebulaGraph。第三步接 reranker 和更强的 LLM-as-judge，提升引用验证准确率。第四步加强 Eval Harness，评估引用准确率、来源多样性、工具成功率和失败原因。
